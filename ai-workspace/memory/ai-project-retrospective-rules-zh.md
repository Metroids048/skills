## 一句话教训

**没锁定「本轮改哪一层问题」就开始改页面** → 越改越多、返工越多、整体感更差。

---

## 通用防踩坑（所有项目）

### 1. 每轮只选一个「主改动类型」

| 类型 | 含义 | 与其它层关系 |
|------|------|--------------|
| 产品主线 | 产品是什么、用户第一步、AI 何时创造价值 | 先于 IA/UI |
| 信息架构 IA | 页面职责划分、导航、信息层级 | 不顺便大改视觉 |
| UI 视觉 | 克制、竞品式、首屏减法 | 不改数据 owner |
| AI/数据闭环 | 上下文、检索、fallback、持久化 | 需先锁 owner |

**禁止同一轮混改。** 「最大权限」≠ 跳过边界确认。

### 2. 输入模糊 → 必须提问

未给出 **目标 + 范围 + 验收** 时，禁止直接写代码。必问：

- 主改动类型（四选一）
- 版本目标：原型 / 内测 / MVP / 商用（四选一）
- 不动清单（页面、数据、测试）
- 验收：完整用户故事 vs 单页（默认完整故事）
- UI 任务：页面验收卡（必须有 / 禁止有 / CTA / 下一步）

配合 skill：`requirement-clarifier` · `workflow-gate` P1。

### 3. 共享层同步收口

改页面时必须一并检查：共享样式、文案、状态逻辑、测试断言。避免「新 DOM + 旧类名 + 旧测试」。

### 4. 数据 owner 先于实现

明确：前端临时态 / 服务端 / 知识底座 / 仅展示。写清输入→存储→AI 读取→写回→降级。

### 5. 验收顺序

完整用户故事 → 单页闸口 → verify 命令 → 才可 claim done。

### 6. 规则必须落文件

「以后别这样」须写入 rules/AGENTS/脚本，不能只在对话里约定。

## AI 求职台产品主线（program1-main 专用摘要）

**用户链路（顺序不可打乱）：**

JD 进入工作台 → 与简历证据打通（解释「为何收到面试」）→ 沉淀问题库 → 模拟面试预演 → 面试中用户主动开实时助手（提词卡，非逐字稿）→ 面试后记录绑定 transcript/提词卡/指标/报告 → 回到问题库复盘。

**产品原则：**

- 首页只服务：**处理当前 JD**
- 实时助手是核心，不是附属
- JD、简历、问题库、搜索进**同一模型上下文**
- 问题库是复习资产，不是训练计划重复页
- 不为对标竞品删掉有效底座功能
- local/DeepSeek/搜索未配置 → **明确 fallback**，禁止伪装 AI 成功

**改产品前 E2E 验收（7 步）：**

1. 首次进入  2. 粘贴真实 JD  3. 诊断能否解释「为何收到面试」  
4. 实时助手突发问题  5. 提词卡引用 JD+简历证据  6. 保存记录与报告  7. 问题库沉淀  

链路不顺 → 禁止堆新功能。
