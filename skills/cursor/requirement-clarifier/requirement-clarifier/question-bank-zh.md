# 需求澄清问题库（中文 · 按场景选题）

> 每轮 **≤5 个**「待确认」；从此库选题，不要照单全问。  
> 已能从仓库/记忆推断的 → 写进 §8 默认建议，别问用户。

---

## 通用（任何 vibe coding 任务可先过一遍）

| # | 问题 | 为何要问 |
|---|------|----------|
| G1 | 做完之后，你希望**第一眼看到什么变化**？ | 把形容词变成可观察结果 |
| G2 | **这次绝对不要动**什么？（页面/功能/数据） | 定边界，防 scope 爆炸 |
| G3 | 有没有**参考**？（竞品、截图、Figma、旧版） | 减审美歧义 |
| G4 | **先交付一小步**的话，哪一块最值得先做？ | 定 MVP |
| G5 | 怎样算「做完了」——给你 demo 还是给可点原型？ | 定交付形态 |

---

## 前端 Demo / 原型（Agent Platform 常见）

| # | 问题 |
|---|------|
| P1 | 改哪些页面？（index / 05 / 运营 11–13 / 全新页） |
| P2 | 只改视觉壳层，还是要改交互/状态流？ |
| P3 | DEMO 数据要不要一起补？还是只改布局？ |
| P4 | 改 `proto.js` 或导航吗？（涉及 ADR-003 须明说） |
| P5 | 验收是否包含 `verify-all` 五步全过？ |

---

## UI/UX 优化（「太丑/太平/不吸引人」类）

| # | 问题 |
|---|------|
| U1 | 主要不满是**布局、配色、字体、密度、动效**哪一类？ |
| U2 | 目标情绪？（专业/活泼/高级/可信…） |
| U3 | 必须保留的品牌/组件规范？ |
| U4 | 移动端 vs 桌面端优先级？ |
| U5 | 是否需要可分享的「卡片态」或截图友好？ |

---

## 新功能 / 新模块（澄清后可能转 zero-to-one-gate）

| # | 问题 |
|---|------|
| N1 | 这是**全新能力**还是现有流程里加一步？ |
| N2 | 用户从哪进、做完去哪？（入口/出口） |
| N3 | 需要新页面还是塞进现有页？ |
| N4 | 状态放哪？（URL / session / 内存 / 后端——原型阶段常用哪种） |
| N5 | 和现有模块有没有重叠？（Agent 应自行查 memory 后复述） |

---

## Agent / 工作流 / 配置类（非页面）

| # | 问题 |
|---|------|
| A1 | 作用在**哪一端**？（Cursor / Claude / Codex / 仓库内） |
| A2 | 是**每次对话**还是**某类任务**才触发？ |
| A3 | 失败时希望**硬拦**还是**软提醒**？ |
| A4 | 要不要写进全局记忆 / ADR？ |
| A5 | 如何验证生效？（新会话？特定 prompt？） |

---

## 选型口诀

```
形容词多（好看、专业、顺滑）→ 优先 G1 + 场景题 U*
只说「做一个 X」           → 优先 N* + G2 + G4
只说「优化一下」           → 优先 G1 + G4 + P1
涉及配置/技能/规则         → 优先 A*
```
