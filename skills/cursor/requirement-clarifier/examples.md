# Examples

## 例 1：模糊 UI 优化（混合型）

**用户：** 小程序初版做完了，移动端 UI 很平，内容不够吸引人，用户没什么分享欲，想整体优化。

**§2 判断：** UI/UX + 前端 Demo 混合；阶段：初版完成后优化。

**§4 不做什么：** 不重写后端；不擅自改核心业务流（除非确认）。

**§7 待确认（示例）：** 是否有设计稿/Figma？核心转化目标是分享还是留存？是否只改视觉还是连带文案与动效？

**§8 默认建议：** 先出 2 个视觉方向 + 关键页（首页/详情/分享卡）再铺开；分享链路优先做「可截图分享」的卡片态。

**§12 执行 Agent：** `frontend-design` + `ckm-design`；先交付方向稿与页面清单，再实现。

## 例 4：Vibe coding — Agent Platform 看板优化（B 类 + Mini-Spec）

**用户：** 帮我把 index 看板优化一下，现在看着太平，信息也不直观。

**§2：** 前端 Demo + UI/UX；阶段：初版后优化。

**§4.5 Mini-Spec（摘要）：**

- 目标用户：内部运营/产品自己点原型  
- 成功标准：index 首屏有 KPI 三卡 + 待办入口明显；`verify-all` PASS  
- 不做：不改 `proto.js` 导航契约；不动 05 页  
- MVP：只改 `index.html` + `style.css` + DEMO 文案  
- 待确认：是否要动侧栏图标风格？  

**§7：** 是否对照某张 Figma？移动端要不要一起改？

**§12：** 执行 Agent 从 Mini-Spec 拷贝范围与验收；先列影响文件再改。
