智能体运营流程文档

本流程文档围绕智能体从注册、审核、上架、运行监控、下架、删除到调用统计的全生命周期展开。文档补齐了 4.1 中隐含的交互闭环：审核驳回后的重新注册路径、上架后的广场与海能Work同步、健康检查失败后的告警与自动下架、已下架后的删除、调用数据进入业务看板。

端到端主流程

1

智能体注册

平台创建、存量迁移或第三方接入先形成统一 agent_id。

2

判断可见范围

private 免审直通；department 和 public 进入审核中。

3

运营审核

检查功能可用性、内容合规、命名规范，并可触发连通性测试。

4

上架分发

审核通过后进入已上架，同步到智能体广场和海能Work可见范围。

5

运行监控

代理调用、采集 usage、统计调用量、成功率、延迟、Token 和错误码。

6

下架/删除/告警

运营可手动上下架；健康检查连续失败可告警并自动下架；仅已下架可删除。

一、注册来源与入口

运营平台采用注册制统一管理所有智能体。来源差异只影响注册方式，注册后的审核、上架、下架、统计流程保持一致。

平台创建

智能体开发平台通过 POST /api/v1/agents/register 自动注册，auto_submit 控制是否注册即提审。

存量迁移

前期通过 SQL 脚本批量导入，包装 OpenAI 兼容协议后纳入运营管理。

第三方接入

运营人员表单注册，系统生成 agent_id 和 API Key，线下安全分发给第三方。

可见范围与审核触发

visibility说明后续流程
private仅创建者本人可见免审核，注册后直接已上架
department指定部门范围内可见进入审核中，运营审核通过后上架
public全平台可见进入审核中，运营审核通过后上架

第三方接入补齐流程

步骤处理人结果
第三方提供智能体信息第三方/运营形成注册表单内容
运营填写注册表单运营人员系统生成 agent_id 和 API Key
线下分发凭证运营人员第三方配置 API Key 校验
运营触发健康检查运营人员/系统通过后可审核上架
健康检查失败系统不允许通过审核，运营联系第三方整改

二、审核与上架交互

审核采用单层运营审核。审核通过即上架；驳回即终结，若需再次上架必须重新注册生成新的智能体 ID。

审核维度

功能可用性、内容合规、命名规范性。第三方智能体审核时应同时完成连通性测试。

通过

状态变更为已上架，智能体进入广场，并按可见范围同步给对应用户。

驳回

写入驳回原因，生命周期终结；如需再次尝试，创建者或运营需重新注册。

审核页面应展示的信息

区域内容用途
基础信息名称、描述、图标、分类、标签、来源、创建者、部门判断内容合规和命名规范
接入信息endpoint、auth_type、capabilities、example_prompts判断服务能力和接入完整性
可见范围private/department/public判断是否需要审核及上架后分发范围

审核操作通过、驳回、填写审核意见推动状态流转

  ├─ visibility = private → 已上架 → 同步广场/海能Work
  └─ visibility = department/public → 审核中
        ├─ 连通性/合规/命名通过 → 已上架 → 同步广场/海能Work
        └─ 驳回 → 终结 → 通知创建方 → 如需再次上架则重新注册

三、运行监控与调用统计

智能体上架后，运营平台作为统一代理入口转发用户对话请求，同时采集统计数据并执行健康检查。

代理调用

用户请求先进入平台统一入口，再由协议适配层转发到智能体 endpoint。第三方智能体携带 Authorization: Bearer {api_key}。

调用统计

平台记录 agent_id、user_id、时间戳、响应延迟、Token usage、成功/失败标记和错误码。

健康检查

审核时可手动触发；已上架智能体每 5 分钟巡检一次。连续失败触发告警或自动下架。

统计指标

指标说明展示位置
调用量按小时/天/周聚合，可按智能体、来源、部门拆分智能体详情、全局统计概览
成功率调用成功次数 / 总调用次数智能体详情、异常分析
平均响应时间/P95/P99衡量接口延迟和长尾性能智能体详情
Token 消耗prompt_tokens + completion_tokens资源管理、成本分析
活跃用户数调用该智能体的去重用户数运营看板
错误分布按错误码聚合 AGENT_UNAVAILABLE、AUTH_FAILED 等异常看板
用户满意度来自对话反馈的赞/踩比例业务看板和反馈分析

健康检查失败处理

失败次数系统行为闭环
连续 1 次失败记录日志保留巡检证据
连续 3 次失败（15 分钟内）平台自动告警通知运营人员运营查看详情并联系负责人整改
连续 6 次失败（30 分钟内）智能体自动下架广场和海能Work同步移除，后续整改后可手动上架

四、生命周期状态流转

4.1 原文包含审核中、已上架、已下架三个核心状态。为交互闭环清晰，这里补充驳回终结和删除终结两个终态。

审核中

department/public 或第三方表单注册后进入，等待运营审核。

已上架

用户可在广场看到并发起对话；平台开始持续统计和巡检。

已下架

从广场移除，但保留注册记录、审核记录和调用统计。

已驳回

审核不通过后的终态，需重新注册才能再次进入流程。

已删除

仅已下架智能体可删除，删除后不再进入运营列表。

注册
  ├─ private → 已上架
  └─ department/public/third_party → 审核中
        ├─ 审核通过 → 已上架
        └─ 审核驳回 → 已驳回（终结，重新注册）

已上架
  ├─ 运营手动下架 → 已下架
  ├─ 健康检查连续 6 次失败 → 已下架
  └─ 更新智能体信息 → 状态不变

已下架
  ├─ 运营重新上架 → 已上架
  └─ 删除 → 已删除

六、接口设计总览

接口按注册、审核、管理、健康检查和统计分组。部分查询接口是根据页面闭环补齐的建议接口。

注册与管理接口

接口方法说明

七、异常与补齐闭环

这些路径是 4.1 实现时最容易遗漏的闭环点，建议在产品和研发实现中显式确认。

审核驳回后怎么办

驳回为终态。创建方收到驳回原因后，若需再次上架，应重新调用注册接口生成新的智能体 ID。

已上架修改是否重审

原设计明确不触发重新审核。为降低风险，建议保留修改历史，并对名称、描述、endpoint 等关键字段进入审计日志。

健康检查自动下架

连续 6 次失败自动下架后，需要同步从广场和海能Work移除，并写入消息中心/告警记录。

删除前置条件

已上架不能直接删除。必须先下架，再删除；删除后统计数据可保留在历史报表中。

第三方 API Key 泄露或失效

运营可重置 API Key，新 Key 仅展示一次，旧 Key 立即失效；第三方需重新配置。

调用失败给用户什么反馈

统一返回 AGENT_UNAVAILABLE、CONTENT_REJECTED、RATE_LIMITED、AUTH_FAILED 或 INVALID_REQUEST 对应降级提示。

闭环补齐建议
1. 审核通过后：写上架时间 → 同步广场/海能Work → 通知创建者。
2. 审核驳回后：写驳回原因 → 通知创建者 → 重新注册才能再次提审。
3. 手动下架后：移除广场入口 → 停止用户新调用 → 保留统计。
4. 自动下架后：记录健康检查失败链路 → 通知运营 → 修复后手动上架。
5. 删除后：运营列表不再展示 → 历史统计仍可按归档口径查询。

闭环检查结论

4.1 智能体运营的主流程已经完整；本 HTML 文档补齐了页面交互、隐含终态、健康检查异常、同步分发、统计回流和建议接口。实现阶段重点关注审核驳回通知、自动下架同步、API Key 重置安全、统计数据留存和操作审计。
